# Test Vectors

This directory contains **normative test vectors** for the I3C-EX
specification. Any implementation claiming conformance to a given
specification version MUST produce bit-exact output for every vector
in the corresponding vector set.

## Layout

```
vectors/
├── README.md                       This file
├── v0.1.0-draft/                   Vectors for I3CEX-0.1.0-draft.md
│   ├── framing/
│   │   ├── preamble/
│   │   └── tlv/
│   ├── envelope/
│   └── ...
└── v1.0.0/                         Vectors for I3CEX-1.0.0 (future)
```

Each vector is a JSON file with the following structure:

```json
{
    "id": "envelope.basic.0001",
    "description": "Minimal EX-1 envelope with sequence=0, timestamp=0.",
    "spec_version": "0.1.0-draft",
    "spec_section": "6.1",
    "inputs": {
        "sequence": 0,
        "timestamp_ns": 0,
        "payload_hex": "deadbeef"
    },
    "expected": {
        "wire_hex": "..."
    }
}
```

## Conformance Testing

The test runner under `tests/unit/test_vectors.py` (to be added) loads
every JSON file in this directory and asserts that the reference
implementation produces the expected wire output for each vector.
Failures block CI.

## Why Not Generate Vectors From the Implementation?

Vectors are hand-authored or derived from the specification examples,
**not** generated by the reference implementation. This prevents
circular validation: if the implementation defines its own truth, the
vectors provide no additional guarantee.

Vectors are immutable once their specification version is frozen. New
vectors can be added under a new specification version; existing
vectors cannot be edited.
